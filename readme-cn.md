# UIQA 技能中文 README

用于 SaaS、ERP、CRM、业财税后台与设计交付验收的 UI 质量检查技能说明

> **文档用途：**这是一份独立的 Word 版 README，供团队阅读、培训、评审和后续修改使用；不替代技能运行时的 `SKILL.md`。

## 1. 技能概览

UIQA 是一个面向 B 端产品和设计交付的质量验收技能。它的核心目标不是做泛泛的审美点评，而是找出会影响用户完成任务、研发落地、响应式稳定性和业务信任的问题。

| 项目 | 说明 |
|---|---|
| 技能名称 | `uiqa` |
| 主要场景 | SaaS、ERP、CRM、业财税后台、管理系统、AI 工作台、设计稿到代码交付验收 |
| 核心产出 | 验收结论、交付评分、UX 风险、P0/P1/P2 问题、修复建议、HTML 报告、Jira/飞书任务 |
| 默认模式 | 完整页面或原型使用 Standard Mode；单屏或小范围问题使用 Quick Mode |

## 2. 适合什么时候使用

- 需要验收一个线上页面、内测页面、本地页面或 Figma 原型是否达到交付质量。
- 需要比较 Figma 设计稿和实际实现之间的布局、组件、状态、业务体验差异。
- 需要检查响应式布局、表格/筛选/弹窗/抽屉/AI 面板等组件是否稳定。
- 需要给研发或设计输出可执行的问题清单和修复优先级。
- 需要把 UI 问题整理成 HTML 报告、Jira 任务或飞书任务。

## 3. 验收关注点

| 检查维度 | 关注问题 | 典型风险 |
|---|---|---|
| 响应式 | 桌面、笔记本、平板、窄屏下是否可用 | 右侧面板裁切、表格操作不可见、按钮组换行异常 |
| 布局结构 | 是否使用稳定的 flex/grid/Auto Layout 结构 | 固定宽度、绝对定位漂移、动态内容撑破容器 |
| 间距与层级 | 页面节奏、卡片内边距、控件分组是否一致 | 层级弱、控件贴边、信息扫描效率低 |
| 字体与信息层级 | 标题、指标、表格、说明文字是否形成清晰梯度 | 关键数据不突出、文字截断、中文孤字换行 |
| 组件与状态 | 按钮、筛选、表格、弹窗、空/错/加载/权限态是否完整 | 核心状态缺失、危险操作无确认、筛选不可恢复 |
| 业务 UX | 页面是否支持目标用户完成真实任务 | 业务场景不匹配、核心动作不可达、AI 输出缺少依据 |
| 信任与可访问性 | 风险证据、AI 解释、语义、对比度、键盘/读屏体验 | 业务决策缺少来源、图标乱码、隐藏控件暴露 |

## 4. 审查模式

| 项目 | 说明 |
|---|---|
| Quick Mode | 检查一个屏幕、小范围改动或单个组件；主要输出 P0/P1/P2 和最高优先级修复建议。 |
| Standard Mode | 检查完整页面或原型；包含评分、风险等级、角色视角摘要和 QA 报告。 |
| Audit Mode | 检查完整工作流、候选发布版本或设计到代码交付；包含证据、评分拆解、任务单和复验标准。 |
| Regression Mode | 比较新旧状态；重点检查新增的布局、交互、响应式和业务状态回归。 |

## 5. 标准执行流程

1. 盘点验收对象：页面名称、业务领域、页面类型、目标用户、来源材料、视口证据和实现上下文。
2. 选择审查模式和规则文件：加载核心规则、评分规则、业务领域规则和所需模板。
3. 确认业务场景：租户、账号、角色、日期范围、数据状态、权限和核心流程是否正确。
4. 按顺序检查：响应式、布局结构、间距、字体层级、表面样式、组件状态、业务 UX。
5. 标注严重级别：所有问题都必须包含影响、根因、修复建议和复验标准。
6. 评分和定风险：完整页面使用 100 分制；存在 P0 时分数上限为 70。
7. 输出交付物：默认生成本地 HTML 报告；如需要，生成 Jira/飞书任务。
8. 交接证据：说明已检查视口、未验证范围、最终评分、风险等级和修复顺序。

## 6. 严重级别定义

| 级别 | 定义 | 示例 |
|---|---|---|
| P0 Blocker | 阻断发布或阻断核心任务完成 | 业务场景错误、核心流程缺失、严重遮挡、主按钮不可用、危险操作无确认 |
| P1 Must Fix | 显著影响交付质量、用户效率或业务信任 | 重要内容被裁切、响应式不可用、AI 输出无证据、关键状态缺失 |
| P2 Recommended | 局部体验和视觉 polish 问题 | 间距不一致、hover/focus 弱、局部对齐偏差、说明文案可优化 |
| Suggestion | 不影响发布质量的可选优化 | 局部效率提升、补充说明、微交互增强 |

## 7. 输出内容规范

一次标准 UIQA 审查建议包含以下内容：

- 结论：Pass / Conditional Pass / Fail。
- 交付评分和 UX 风险：Low / Medium / High / Critical。
- P0/P1/P2 问题表：模块、问题、证据、影响、根因、修复建议、复验标准。
- 业务 UX 评估：页面是否支持目标用户完成核心任务。
- 响应式与布局评估：检查过的宽度、裁切、溢出、表格或表单表现。
- 组件与状态覆盖：按钮、筛选、表格、导航、空/加载/错误/权限、弹窗/抽屉、AI 状态。
- 修复优先级：先修阻断问题，再修发布风险，再补状态和 polish。
- 本地 HTML 报告路径或可访问 URL。

## 9. 示例提示词

- 使用 `$uiqa` 验收这个 Figma 设计稿和线上页面，输出 HTML 报告。
- 用 UIQA 检查这个后台页面的响应式和组件状态，重点看 1024 和 768 宽度。
- 对这个 PR 做 UI 交付质量审查，只列 P0/P1/P2 和修复建议。
- 帮我把这些 UIQA 问题整理成飞书任务，每个问题一个任务。
- 对比旧版和新版页面，找出响应式、表格和筛选交互的回归问题。

## 10. 使用注意事项

- 优先检查业务任务是否能完成，再检查视觉 polish。
- 不要在没有目标设计或页面证据时声称像素级还原。
- 如果没有移动端证据，要明确写出“未验证移动端”。
- 评分必须受证据约束：有 P0 不应超过 70 分，响应式或重要状态未验证不应超过 85 分。
- AI 输出、财税风险、审批或合规类信息必须关注来源、证据、置信度和审计追踪。
- 报告要面向产品、设计、研发都能执行，避免只写“视觉不好”这类不可操作描述。

## 11. 维护建议

如果后续继续维护该技能，建议只把运行时必须读取的内容放在 `SKILL.md` 中；详细行业规则、模板和示例继续拆到独立文件，保持渐进式加载。

- 新增业务领域时，在 `business/` 下增加规则文件，并在 `SKILL.md` 的资源路由中说明何时读取。
- 调整评分标准时，同步更新 `docs/scoring_system.md` 和示例报告。
- 如果默认输出 HTML 报告，要保持模板自包含、无外部 CDN、可离线打开。
- README 属于给人看的说明材料，不应作为技能运行时依赖；真正的执行规则仍以 `SKILL.md` 和引用文件为准。

## 12. 一句话总结

> **总结：**UIQA 是一个面向 B 端复杂业务界面的交付验收技能，重点不是评价好不好看，而是判断页面是否能稳定、可信、高效地支持真实业务任务。
